Strategy Design

What actually works for these 8 high-volatility tech/finance stocks
💡
Recommended strategy: EMA Crossover + RSI Filter. This is a classic momentum strategy that works well with NVDA/TSLA/META-style volatile stocks. It's simple enough to implement cleanly but defensible enough to explain. Avoid overly complex ML strategies — they overfit and are hard to reproduce.

EMA 9 / EMA 21 Crossover with RSI Confirmation

Timeframe
Check every 5 minutes during market hours (9:30 AM – 4:00 PM ET)
Buy signal
EMA9 crosses above EMA21 AND RSI(14) is between 40–65 (momentum, not overbought)
Sell signal
EMA9 crosses below EMA21 OR stop-loss hit OR take-profit hit
Capital per trade
12.5% of portfolio per position (8 stocks = 100% max deployed)
Stop-loss
-3% from entry price — hard exit, no exceptions
Take-profit
+6% from entry price (2:1 risk/reward ratio)
Max open positions
4 simultaneous positions maximum (50% capital deployed at peak)
API failure
Log error, skip this cycle, do NOT close existing positions. Retry next tick.
Missing data
Skip that ticker for this cycle, log a warning. Never trade on incomplete signal.
Indicators
Computed via pandas-ta or ta-lib — run on last 50 candles of 5m data
⚠️
Pattern Day Trader rule: If using a real account with under $25,000, you're limited to 3 day trades per 5-day rolling period. Paper trading has no PDT restrictions — but document this as a "what's needed before going live" requirement.

Key design principle: strategy is environment-agnostic

The SignalEngine takes a DataFrame and returns a signal. It doesn't know or care whether the data came from yfinance (backtest) or Alpaca live feed. This is what makes backtests trustworthy — the exact same function runs in both modes.

Backtesting Approach

How to produce results they can't dispute
🚫
Do NOT use backtesting libraries as a black box. Using Backtrader or Zipline as-is makes it hard to prove that your live bot uses the same logic. Write a custom backtester (it's ~150 lines) using the same SignalEngine functions — this directly satisfies "same strategy logic must be used across all timeframes."

Custom vectorized backtester approach

  • Download 3yr history per ticker via yfinance in one shot, resample to 5-min bars
  • Slice the DataFrame for each period (1wk, 1mo, 3mo, 6mo, 1yr, 3yr) — same data, different window
  • Run SignalEngine.generate_signal() on each row in sequence (no lookahead bias)
  • Simulate fills at next-bar open (realistic — not at signal candle's close)
  • Track position, capital, drawdown, fees (assume $0.005/share or 0.1% slippage as assumption)
  • Output: CSV + summary dict with all required metrics per period

Risks & Gaps in a Naive Approach

Where most submissions fail this kind of task
🚫
Lookahead bias — Using future data in signal computation. The most common mistake: computing EMA on the full dataset, then checking if it crossed at time T. The correct approach: at each timestep T, only use data up to T.
🚫
Survivorship bias — All 8 tickers are mega-caps that already survived and grew. Backtests on these will look good regardless. Acknowledge this explicitly in your research document.
⚠️
Overfitting parameters — Don't tune EMA(9,21) and RSI(14) by running backtests and picking whatever made the most money. Choose parameters before seeing the results, then run the test once.
⚠️
Missing market hours check — Your bot must check that the market is open before placing orders. Alpaca has an /v2/clock endpoint — use it at the start of every cycle.
⚠️
Fractional shares — Alpaca supports fractional shares in paper mode. If you don't use them, your 12.5% capital allocation won't be exact for high-priced stocks like NVDA or GOOGL.
✅
What impresses reviewers: Showing a period where the bot lost money (e.g., a sideways/bear market period) and explaining why. It proves you didn't cherry-pick and that you understand the strategy's weaknesses.

Kanban Board Task (Take-Home Assignment)

Frontend Fullstack Intern — separate from the trading bot
ℹ️
This appears to be a take-home assignment from a different company (in your job folder, not the paper-stock-bot folder). It's a solid fullstack frontend task — here's the key approach guidance.

Tech decisions that matter

  • dnd-kit over react-beautiful-dnd — rbd is unmaintained as of 2023. dnd-kit is actively maintained and React 18+ compatible
  • indexedDB via idb library over raw localStorage — handles large task sets, async, and is the "right" choice they're testing
  • shadcn/ui — use it, it shows you know modern React tooling
  • Next.js App Router — use Server Components where possible, but Kanban board itself is client-side

The dnd-kit gotchas

  • Use DndContext at Board level, SortableContext per Column
  • Use DragOverlay for the dragged card — don't move the original DOM element
  • Store the active item in state during drag to render the overlay correctly
  • closestCorners collision detection works better than closestCenter for Kanban layouts
  • Handle cross-column drops in onDragEnd by checking over.id vs column IDs

Bonus feature recommendation: Task count per column

Of the 4 options, "Display task count per column" is the easiest to implement (2 lines), looks great in a demo, and doesn't add complexity that could introduce bugs. Save your time for polish and the video.

✅
What they're actually testing: Clean code + ability to explain decisions. They explicitly say "we value clear communication as much as feature completeness." In your video, don't just demo — explain WHY you chose dnd-kit over alternatives, why idb over localStorage, etc.

Suggested Execution Plan

For the trading bot — ordered to avoid rework
1

Write the Research Document first

Don't skip this. It forces you to understand the space before committing to choices. Cover: Alpaca, Tradier, IBKR, Quantconnect, Backtrader, yfinance. Explain tradeoffs, not just lists.

2

Define and document the strategy

Write down all the rules (buy/sell/stop-loss/position sizing) BEFORE writing any code. This becomes your strategy document and ensures the backtest and live bot use the same rules.

3

Build the SignalEngine module with unit tests

Pure functions only. Takes a DataFrame, returns BUY/SELL/HOLD. Test it on known OHLCV sequences so you're confident before running the backtest.

4

Build and run the backtester

Download yfinance data for all 8 tickers × 3 years. Run the backtest across all 6 periods. Generate the trade log CSV and summary metrics. Fix any logical bugs.

5

Build the live paper trading bot

Connect to Alpaca paper account. Wire up the same SignalEngine. Add order execution, position tracking, SQLite logging. Use APScheduler for the 5-min loop with market-hours check.

6

Write the "going live" checklist

What changes to go from paper to real: change env var, verify PDT compliance, set actual stop-loss orders server-side, add real capital limits. This is deliverable #9 and often overlooked.

7

Package all 9 deliverables

Research doc, platform explanation, strategy doc, bot code, backtest report, trade logs, P&L summary, run instructions, live-trading checklist.

Bottom Line

Use Alpaca for paper trading, yfinance for backtesting data, EMA 9/21 + RSI 14 as the strategy, and keep your SignalEngine as a pure module that both the backtester and live bot share. That single architectural decision is what makes this submission trustworthy and scalable. Tackle backtesting before the live bot — not after.
